# Rocket Ground Station for AGH Space Systems

This project implements a desktop application used for control and monitoring of a rocket fueling, launch and recovery. The app is written in Python and provides a graphical user interface based on a [Kivy](https://kivy.org/#home) framework. 

A headless mode for developers is present in the repository but has not been updated and tested for a long time, currently we recommend against using it.

## Other tools and relevant repositories

- [ParseHub](https://github.com/AGH-Space-Systems/parse_hub "Application parsing electronics debug data, handling visualization and data conversion")

## Features
- Configuration and control of various rocket and ground station onboard devices
- Sequence programming and upload
- Monitoring and logging of all received vehicle telemetry,
- Restarting of hardware boards and selected onboard devices
- Aborting in case of any problems.
- Rocket hydraulics visualization
- Selective arming of each device
- Selective synchronization of each device

## Additional application startup parameters:
In order to use them start application via terminal.

```--mock``` - Start application in communication layer simulation mode, this way application will behave as if it was connected to physical rocket hardware

**Note:** Sensor values generated by the communication mock are randomised, thus they do not reflect the operation of real rocket hardware.

```--debug``` - **Not recommend for production** Runs application in development mode, with this flag protections against application crashes are disabled

## Installation and running

### From prepackaged archive (Recommended method)
Download an application package for a specific application release from repository releases page: https://github.com/AGH-Space-Systems/rocket-ground-station/releases

Application packages have the following naming scheme: 

```rocket-ground-station-win64-seeminglyrandomletters.zip```

example: 
```rocket-ground-station-win64-4090d9a.zip```

Extract the zip archive to a location of your choosing, the extraction location does not matter.

After extracting the zip file, you will have a single directory called ```rocket-ground-station```

In order to start the application enter that directory and double click the file: ```rocket_ground_station.bat```, the same as you would start any other program.

you can also start the application via terminal, for example when you need to supply additional launch flags to the application. You can do it in the following way:

cmd:
```batchfile
cd rocket-ground-station
rocket_ground_station.bat --mock --debug
```
powershell:
```powershell
cd rocket-ground-station
cmd.exe \c 'rocket_ground_station.bat --mock --debug'
```

#### Troubleshooting:
There is a possibility that this you don't have one of python dependencies (C Runtime libraries) installed on your system. In that case please download and install it from the following link:
https://aka.ms/vs/17/release/vc_redist.x64.exe

After you finish installation, try starting ```rocket_ground_station.bat``` again.

If the issue persists please contact one of the software team members.

#### Why use prepackaged archive over a regular git based installation?

In order to provide application in the most stable manner, we have opted to test it against specific
library and python versions. Prepackaged archive is a solution to let users easily obtain exactly the same
python environment as the one application was tested in.

Using such environment greatly decreases likelihood of unexpected issue happening during application usage, thus it is
**greatly recommended** to use it in production scenarios.

#### Can I still launch the application by directly calling main.py from python?

Yes, it is possible but **not recommended**, especially if the user does not have experience with 
python environment management. Prepackaged archives ship with an embedded python installation 
(embedded as in provided with application, not in the context of microcontrollers), thus that specific 
version of python should be used. Using any other python environment could result at 
best in an instant crash of application and at worst with hard to catch errors during application runtime.

If you still wish to proceed, you can launch the application with correct python environment in the following way:
```
cd rocket-ground-station
python_embed\python.exe -m pip install -e .
python_embed\python.exe rocket_ground_station\main.py
```

#### Why use a batch file for startup and not an .exe file?

In the past we have explored using solutions such as ```pyinstaller``` or ```py2exe``` to package the application.
But we reached the conclusion that their features and implementation aren't exactly aligned with our project goals.

The biggest problem is the fact that a python environment must be provided with the application no matter what.
In order to do that, such tools implement various abstraction layers to hide this dependency away from the user.
Mostly it results in python env being extracted to ```%temp%``` upon application startup and deleted after closing or
simply keeping the environment in some additional tool specific folder structure right next to the application's code.

So for what exactly is that .exe file, since python is running the application and it's not 
getting compiled or anything like that? Well it's used to setup execution environment so that users pc executes code
with correct python env, and sometimes to unpack whole python environment to ```%temp%``` and perform other additional
startup and shutdown procedures. Additional abstraction layer like such .exe can cause side effects 
which might impact application stability.

Moreover, it can cause problems with keeping the application code editable, since there is always a possibility of
needing to perform emergency fixes on site in a production scenario. Most of the application code is actually 
a library, which needs to be installed as an editable package in order work without reinstalling it after each change
in the code.

Our method of packaging the application combined with .bat file startup incorporates elements we found 
favorable in aforementioned packaging tools, while fulfilling our project requirements by providing a user system
independent python installation with correct version of each tools in the environment.

As such, our solution provides better user experience for this project than other packaging tools would
while not impacting negatively the applications stability


### From source (development installation)
**Prerequisites:**
- Supported python versions: 3.7 - 3.12
- Operating systems: Windows (10, 11) 

**Note:** Please don't report issues for configurations not fulfilling above requirements.

**Note:** Linux based systems work but are not recommended for production environment. MacOS is fully unsupported though it might provide partial functionality. Please don't report issues for MacOS.

Please clone this repository in your directory of choice by running

```
git clone https://github.com/AGH-Space-Systems-Software/rocket-ground-station.git
```

Make sure you have a supported version of Python installed. Then go to the root project directory and run

```
python3 -m pip install -e .
```

This should install the project in a develop mode and update all other dependencies.
To run the application go to the root project directory and run

```
python rocket_ground_station/main.py
```

In case of any problems with permissions to serial port on Linux use:
```
sudo usermod -a -G dialout $USER
```
## Development

### Contributing

```main``` is the main branch of the repository, ```master``` and ```develop``` are unused and not updated.

In order to contribute please create a new branch based on latest ```main``` commit and create a pull request.
No commits should be ever directly pushed to ```main```. If your pull request is not yet ready for review
please add a ```WIP``` tag to it. Once you finish working on it remove the tag and request a review from someone.
Keep in mind the pull request template and fulfill checks presented there (if applicable) before requesting a review,
since the template contains things that can be easily omitted while working on a feature.

Each pull request should pass a pylint check before being merged.
If you'd like to, you can run pylint code checks locally before pushing to a branch:
```
pylint --output-format=text --rcfile="./workspace/.develop_pylint_config" rocket_ground_station
```
Pull requests should always be merged by squashing.

While developing make sure to always use ```--debug``` flag in order to detect all potential crash sources.

All pull request should be tested before merging with rocket communication layer simulator
provided by either ```--mock``` flag, ```tools/standalone_serial_mock.py```, ```tools/standalone_tcp_mock.py```
or tested with actual rocket and ground station hardware if they are available.

Using standalone mocks is recommended since ```--mock``` flag replaces a part of applications communications handling
and can lead to behavior mismatches.

In order to start standalone TCP mock, run the following programs:

Simple TCP proxy
```batch
# Simplified version of rocket_proxy usable for development purposes
python tools\simple_tcp_proxy.py --tcp_address 127.0.0.1
```
Standalone TCP Mock
```batch
# Feed interval governs how often feed frames will be sent, the value is a float with second as it's unit
python tools\standalone_tcp_mock.py --proxy_address 127.0.0.1 --feed_interval 1
```

### Project structure

Currently, rocket_ground_station (Grażyna) is the main way of communicating with the ground station
and by extension of that, with the rocket. Grażyna is made to primarily be extensible and support as many different
rockets as possible. This way only rocket specific features need to be developed between projects.
In the past year (2023 - 2024) the application has undergone multiple changes, such as: arming system changes,
vast performance improvement, Hydraulics visualization, multiple transport type support and more.

The application itself can be split into two main components: 
- The UI components (contained in ```components``` and ```layout``` directories)
- Core library code (contained in ```core```)

Application UI was made in ```kivy```, this framework mostly fulfills our requirements but has some quirks and issues
that sometimes make development quite troublesome. Future software projects likely will use different library and tech stack
depending on their requirements.

Core library code contains actual logic used to control the rocket and the ground station, like specific device handling classes
and communication management. Core library can be used outside of Grażyna, and likely will be introduced in other software projects.

The project directory structure is as follows:
```yaml
rocket_ground_station: <-- Main application directory
   assets <--various static assets packaged with the application
   components <-- Application UI component logic
   configs <-- Application configuration files, both app specific config and Hardware configuration files
   layouts <-- Application UI component layout (.kv language files)
   core <-- Application Core library
   logs <-- Application output logs
   tools <-- Various Grażyna specific tools useful during development 
```

### Common regressions to watch out for during development
- **Crashes during hardware config changes**. Since hardware configs govern a lot of application functionality,
their reloading is a commonplace operation during application usage. Moreover, loading of default config during startup
varies from loading a new one during runtime, thus issues that didn't occur before can happen after config reload.
Remember that config currently set as default might not contain all devices supported by the application.
If code you are working on introduces application wide changes, remember to test it with all device types 
present in the config.

- **Crashes after disarming a device and trying to perform an action on it**. Some operations can be executed on devices
outside of their respective device widget, for ex.: Dynamixel resets, Hydro actions. These operations often expect
device to be armed and if it's not, they perform an implicit arming before continuing. But sometimes it can be handled
improperly, thus remember to always arm and then disarm all devices to test such functionalities.

**Remember to always thoroughly test the feature or fix you are developing**
Try to break it deliberately, because if you don't find the issue first, users certainly will stumble upon it later.


## Feature notes

### Hydraulics visualization

In order to use hydraulics visualization you need to have activated an appropriate hardware configuration.
Currently, visualization is configured to work with Turbulence rocket hardware configurations. You can still use the application with hardware configurations for other rockets, but hydraulics visualization will not work for them.

Example of configuration files supported by the visualization:
```trb_27.10.yaml, trb.yaml```

Moreover, a specific hardware configuration file has to have appropriately mapped its devices to the visualization's components.
You will be able to provide such mapping in hardware configuration file by setting a ```hydro_name``` property for each device taking part in visualization.

Below you can see an example of addition of such property to a device in HW config.

```yaml
devices:
  dynamixel:
    Fill Ox:
      hydro_name: 'FOV'   # <---
      board: kromek
      closed_pos: 1200
      device_id: 4
      open_pos: 3350
```

In case you would like to see all possible names to which you can map devices, please refer to the 
```rocket_ground_station/layouts/hydro_visualisation_trb.kv``` file.
Each entry contains a ```name``` property which is the name you can map to via ```hydro_name``` properties in HW configs.

```
ValveVisual:
        name: "FOV"
        icon_center_pos_hint: (2095/3147, 1-752/1440)
        button_placement: "top"
        icon_rotation: "horizontal"
```

In case there is a need for additional devices or telemetry being visible in the hydraulics tab, they can be easily added to the visualization layout.
To have any of the devices visible in the hydraulics tab, an appropriate ```CardVisual``` class entry needs to be added to the ```rocket_ground_station/layouts/hydro_visualisation_trb.kv``` file.

```
CardVisual:
  name: "scheduler_visual" <-- Used for mapping to the device in hardware tab
  resize: True
  size_hint: (.15, .15)
  pos_hint: {"center_x": 0.085, "center_y": .73} <-- Determines the position of device in hydrualics tab, minimal value for a coordinate is 0 and max is 1. Coordinates are oriented left to right (x) and bottom to top (y).
```

Next the CardVisual instance needs to be mapped to the actual device instance

```yaml
scheduler:
  Scheduler:
    hydro_name: 'scheduler_visual' <-- Needs to be the same as name in CardVisual entry.
    board: broadcast 
    close_on_finish: false
    device_id: 0
    expect_acks_from:
    - kromek
```

### View Profiles

You can define profiles in hardware configurations that will let you filter device visibility in the hardware tab
This way you can prepare views for specific parts of interaction with rocket hardware.
For example a separate view for cold flow testing containing only devices most commonly used during this procedure.
A view profile is structured in the following way:
```yaml
view_profiles:
  - profile_name: Profile_1
    device_names:
      - Fill Ox
      - Chamber
  - profile_name: Profile_2
    device_names:
      - Manifold fuel(wika black)
      - Load Ox
```

The above snippet was taken from ```trb_27.10.yaml``` config file, which incorporates these example profiles.

Active profile can be changed using the drop down in the upper right corner of hardware tab (by default with text "All Devices"), right next to active HW configuration dropdown.

### Device sync buttons

The application provides an ability to selectively synchronize devices, such action can be performed using a round arrow button accessible in the upper right corner of each device card.

The button itself changes color depending on the result of last operation performed during current application session:

- White: No synchronization was attempted
- Green: Last synchronization was successfull
- Red: Last synchronization was unsuccessfull

### Device arming buttons

Devices in the application can be armed selectively using small lock shaped buttons located in the upper right corner of each armable device found in hardware tab.

The buttons themselves have two states:

- The lock icon is closed: Device is disarmed
- The lock icon is open: Device is armed

### ACK/NACK indicator icons

Each device that makes use of service action type has an ACK/NACK indicator icon next to the button performing that specific action. The icon itself is represented by a small circle.

The icon can have 3 possible states, signified by it's color:

- Greyed out: Operation was not performed since the configuration has been loaded or the device has been synchronized (both of these operations grey out the icon)
- Green: Last performed operation was successful
- Red: Last performed operation was not successful

### Heating Lamp Automation

The application provides automation for pressure control using heating lamps. The heating lamp monitors a pressure sensor and automatically turns on or off to maintain the target PV pressure:

- The lamp turns on when pressure falls below (target - margin)
- The lamp turns off when pressure rises above (target + margin)

#### Safety Features
- When a heating lamp device is disarmed, automatic control is immediately disabled and the lamp turns off
- When automatic control is manually disabled, the lamp always turns off as a safety precaution
- Visual indicators show when pressure is outside the target range

#### Pressure Display Colors
The current pressure display changes color based on its proximity to the target:
- Within margin range: Primary light color (light green)
- Close to target but outside margin: Accent color (yellow orange)
- Far from target: Red

#### Configuration
Heating lamps can be configured in hardware configuration files as follows:

```yaml
heatinglamp:
  LampName:
    sensor_name: "Name of Sensor Device"  # Name of the sensor to monitor
    target_pressure: 50.0                 # Target pressure to maintain
    margin: 0.5                           # Acceptable deviation from target
    board: kromek
    device_id: 0                          # Since heating lamp inherits from relay, make sure to not overlap ther device IDs!
```

Example configuration is created in ```trb_27.10.yaml```

**Important note!**

HeatingLamp and Relay device ID ranges overlap (since heatinglamp is made to work with relays) so make sure to keep their ```device_id``` properties unique among those device types. If one relay uses a specific device_id, then no other heating lamp or relay should use it. Otherwise unexpected behavior in the application might emerge.

The `sensor_name` must match the name of an existing sensor device in the configuration.

#### Operation Modes
- Manual mode: Operator can manually turn the lamp on or off
- Automatic mode: The system regulates the lamp based on sensor readings and target pressure

### Hardware configuration parameters

#### Data types
Data types on HW configs are case-insensitive, float is used 99% of time for sensors.

- NO_DATA
- UINT32
- UINT16
- UINT8
- INT32
- INT16
- INT8
- FLOAT
- INT16X2
- UINT16INT16

#### Board names
Board names are case-insensitive

- GRAZYNA
- STASZEK
- RADEK
- CZAPLA
- PAUEK
- KROMEK
- ANTEK
- OLA
- LAST_BOARD
- BROADCAST

#### Data Filters
- IdlePass - Lack of filtering, data is passed without any modifications to display widgets.
Arguments: 
- - None
- MovingAverage - Applies moving average over last N values of a sensor. 
Arguments:
- - buffer_length: Int - Last N values of a sensor from which average will be calculated.
- CustomExpression - Applies a custom mathematical expression to transform sensor data. The expression is evaluated with the current sensor value as 'x' and has access to the math module.
Arguments:
- - expression: Str - Mathematical expression to apply, where 'x' represents the current sensor value. Example: "2 * x + 5" or "math.pow(x, 2)"

**Note:** CustomExpression uses eval() internally for expression evaluation. While functional, use with caution and ensure expressions don't perform harmful operations. 

Devices specified in the hardware configuration files can be configured using the following parameters:

#### All devices
- board: Str - Board that controls this specific device
- device_id: Int - Id of device instance, unique to its device type
- snapshotting: Bool (Optional, default False) - Whether device should be included in snapshots (currently only sensors and pistons are supported)
- hydro_name: Str (Optional, default "") - Name of the device on the Hydraulics visualization

#### Dynamixel
- open_pos: Int - Open position of the dynamixel
- closed_pos: Int - Closed position of the dynamixel
- Velocity: Int - Velocity of movement of the dynamixel

#### Flash
- None

#### Igniter
- burntime: Int - (Unconfirmed) time in seconds during which igniter should be lit

#### MultiSensor
- (+ all parameters available in Sensor devices since MultiSensor is a subclass of Sensor)
- dependencies:
  - aggregate: List[str] - List of names of sensor devices that this MultiSensor should aggregate

#### Parachute
- None

#### Sensor
- data_type: Str - Data type of the sensor, data types can be seen in a dedicated table. Usually it's a float
- recording: Bool - Currently not functional
- units: Str - Units of data shown in sensor widget
- scale: Float - Value by which shown sensor data will be scaled
- a: Float - Calibration coefficient, resulting sensor value (where X is value and S is scale) is: aXS + b
- b: Float - Calibration coefficient, resulting sensor value (where X is value and S is scale) is: aXS + b
- round_to: int (Optional, default -1) - Number of decimal places to which the sensor value should be rounded, 
-1 disables rounding completely
- data_filter_name: Str (Optional, default "IdlePass") - Case insensitive name of data filter that should be
used by the sensor, filter is applied after scaling and calibrating a value. Available filters are described in the
data filter table.
- data_filter_args: Dict (Optional, default {}) - Arguments passed to the filter instance. 
Described in data filter table.

#### Piston
- (+ all parameters available in Sensor devices since Piston is a subclass of Sensor)
- min_val: float - Value of minimal piston extension
- max_val: float - Value of maximal piston extension
- volume: float - Volume of the tank in which piston is contained
- length: float - Length of the tank in which piston is contained
- volume_units: Str (Optional, default "ml") - Units in which volume value was provided
- length_units: Str (Optional, default "mm") - Units in which length value was provided
- display_volume: Bool (Optional, default True) - Should volume information be displayed
- display_length: Bool (Optional, default True) - Should length information be displayed
- max_percentage_safe_val: float (Optional, default 100) - Max safe value of piston position, used for 
showing warning when piston moves out of safe area
- min_percentage_safe_val: float (Optional, default 40) - Minimal safe value of piston position, used for
showing warning when piston moves out of safe area
- show_fuel_volume: bool (Optional, default False) - Whether to show oxidizer's or fuel's volume in the widget

#### Recovery
- None

#### Relay
- is_reversed: Bool (Optional default False) - Whether relay operations should be reversed, 
for ex.: close will send open frame

#### Reset
- None

#### Scheduler

- excpect_acks_from: List[str] (Optional, default None) - Board names from which scheduler acks are expected

#### Servo
- open_pos: Int - Open position of the servo
- closed_pos: Int - Closed position of the servo

#### Supply
- None

#### HeatingLamp
- dependencies:
